# 事件循环

## 【问题】
解释 JavaScript 的事件循环机制

## 【回答】
JavaScript 是单线程语言，通过事件循环实现异步。

**执行顺序：**

1. 执行同步代码（主线程）
2. 遇到异步任务，交给相应模块处理（如定时器、I/O）
3. 异步任务完成，回调放入任务队列
4. 主线程空闲时，检查任务队列并执行回调

**任务队列：**

1. 宏任务：`script`、`setTimeout`、`setInterval`、`setImmediate`、I/O、UI 渲染
2. 微任务：`Promise.then`、`process.nextTick`、`MutationObserver`

**执行规则：**

1. 执行一个宏任务（从宏任务队列）
2. 执行所有微任务（清空微任务队列）
3. 如有必要渲染 UI
4. 重复上述过程

## 【问题】
setTimeout(1000) 一定是 1000ms 吗？为什么？和 RAF 的对比

## 【回答】
setTimeout(1000) 并不保证一定在 1000ms 时执行，它表示至少等待 1000ms 后才会进入任务队列，真正执行还要等当前 JavaScript 执行栈清空并轮到事件循环调度，如果主线程被阻塞（如大量计算、同步代码），回调就会延迟执行；此外浏览器还有最小时间间隔限制（clamp），嵌套定时器可能被强制延长到 4ms 以上。相比之下，requestAnimationFrame 会在浏览器下一次重绘前执行，通常与屏幕刷新率（如 60Hz ≈ 16.7ms）同步，因此更适合动画场景，可以避免掉帧并减少不必要的计算。

## 【问题】
定时器不准确的原因是什么？

## 【回答】
定时器不准确主要有几个原因：第一是**事件循环机制**，定时器回调需要等待当前任务执行完才能进入执行阶段；第二是**主线程阻塞**，如果有长时间同步任务会推迟回调执行；第三是**浏览器节流策略**，例如后台标签页或隐藏页面会降低定时器频率；第四是**最小延迟限制（timer clamping）**，浏览器为了性能和安全会限制最小时间间隔，因此 `setTimeout` 更像是一个延迟触发机制而不是精确计时器。

## 【问题】
为什么 Promise 可以解决回调地狱？

## 【回答】
回调地狱的问题在于**多层嵌套回调导致代码结构混乱、难以维护、错误处理分散**。Promise 通过**链式调用（then）和状态管理（pending / fulfilled / rejected）**把异步流程变成**扁平化结构**，每一步异步操作返回一个新的 Promise，从而可以顺序组织逻辑；同时 Promise 还提供统一的 `.catch()` 进行错误处理，使得异常可以沿链向下传递，因此代码可读性和可维护性大大提升。

## 【问题】
Promise 链式调用是如何实现的？

## 【回答】
Promise 链式调用的核心在于 **then 方法会返回一个新的 Promise 对象**。当上一个 Promise 状态变为 fulfilled 时，会执行 then 的回调函数，并根据回调的返回值决定下一个 Promise 的状态：如果返回普通值，下一个 Promise 会被 resolve；如果返回 Promise，则会等待该 Promise 状态再继续链式传递；如果抛出错误，则会进入 reject 状态并被后续 `.catch()` 捕获。正是这种**返回新 Promise + 状态传递机制**实现了链式调用。

## 【问题】
为什么会出现内存泄漏？JS 中有哪些情况会造成内存泄漏？

## 【回答】
内存泄漏本质是指**不再需要的对象仍然被引用，导致垃圾回收器无法回收**。在 JavaScript 中常见的情况包括：**未清除的定时器（setInterval / setTimeout）、未移除的事件监听、闭包长期持有外部变量、全局变量滥用、DOM 引用未释放、以及缓存或 Map/Set 持续增长**等。例如闭包中引用了一个大对象，即使函数执行结束，只要闭包仍被引用，该对象就不会被 GC 回收，因此在实际开发中需要及时清理引用或使用 `WeakMap` / `WeakSet` 等弱引用结构来避免泄漏。
