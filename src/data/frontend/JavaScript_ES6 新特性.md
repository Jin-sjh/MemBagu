# ES6+ 新特性

【问题】列举 10 个 ES6+ 的重要新特性

【回答】

块级箭模解，默展类模诺，符步链空大，全局

1. **let/const** - 块级作用域变量声明

2. **箭头函数** - `() => {}` 简洁语法，无自己的 `this`

3. **模板字符串** - \`Hello ${name}\` 支持插值

4. **解构赋值** - `const {a, b} = obj`

5. **默认参数** - `function(a = 1) {}`

6. **展开/剩余运算符** - `...arr` / `function(...args)`

7. **类语法** - `class` 关键字

8. **模块化** - `import/export`

9. **Promise** - 异步编程解决方案

10. **Symbol/Map/Set** - 新的数据类型

11. **async/await** - 更优雅的异步处理

12. **可选链** - `?.` 安全访问属性

13. **空值合并** - `??` 提供默认值

14. **BigInt** - 大整数支持

15. **全局对象标准化** - `globalThis`

16. 手写代码题

## 【问题】
CommonJS 与 ESModule 的核心差异是什么？

## 【回答】

| 内容 | CommonJS | ESModule |
|------|----------|----------|
| 导入语法 | require | import |
| 导出语法 | module.exports / exports | export |
| 加载方式 | 运行时同步加载 | 编译时静态分析 |
| 导出值 | 值拷贝 | 实时绑定 |
| 顶层作用域 | 无法在顶层分析 | import/export 提升 |
| 循环依赖 | 可能未初始化 | 始终绑定 (更安全) |
| 浏览器支持 | 需工具处理 | 原生支持 |
| Tree Shaking | 不支持 | 可实现 (依赖静态分析) |

ESM 更适合现代工程优化 (如摇树优化、代码分割、预加载)。

## 【问题】
CommonJS 和 ES Module 的差异？

## 【回答】

**核心一句话总结**

CommonJS 是**运行时动态加载**的 Node.js 原生模块化方案，ES Module (ESM) 是**编译时静态分析**的浏览器 / JS 标准模块化方案，二者在加载时机、语法、导出导入特性上有本质区别。

**详细对比（面试分点答）**

| 对比维度 | CommonJS (CJS) | ES Module (ESM) |
|----------|----------------|-----------------|
| **加载时机** | 运行时加载（代码执行到 `require()` 时才加载模块） | 编译时静态加载（代码执行前就分析依赖，确定所有导入导出） |
| **语法规范** | `require()` 导入，`module.exports` / `exports` 导出 | `import` / `export` 关键字，支持默认导出 `export default` |
| **导出特性** | 导出的是**值的拷贝**（基本类型隔离，引用类型共享引用） | 导出的是**值的引用**（实时绑定，原模块修改会同步到导入方） |
| **循环依赖处理** | 基于缓存，导出未完成的 `exports` 对象，可能拿到不完整的值 | 基于实时绑定，能正确处理循环依赖，保证值的同步 |
| **Tree-shaking 支持** | 不支持（动态结构无法静态分析） | 天然支持（静态结构可被打包工具分析，移除未使用代码） |

**关键原理补充**

- **值的拷贝 vs 引用**：
  - CommonJS: `module.exports = { a: 1 }` 导出后，导入方拿到的是这个对象的**浅拷贝**，如果原模块修改 `a`，导入方不会感知；
  - ESM: `export let a = 1` 导出后，导入方拿到的是 `a` 的**实时引用**，原模块修改 `a`，导入方会立刻拿到最新值。

- **为什么 ESM 是标准**：ESM 是 JS 语言层面的模块化标准，统一了浏览器和 Node.js 的模块化方案，而 CJS 是 Node.js 社区的方案，仅服务于服务端。

## 【问题】
从 CommonJS 转向 ESM 的迁移策略是什么？

## 【回答】

迁移顺序可参考：

1. **保留旧代码，用 .cjs 标记** - 保持原有 CommonJS 代码可用

2. **新代码使用 ESM** - 新增代码直接采用 ES Module 规范

3. **配置 "type": "module"** - 在 package.json 中声明模块类型

4. **将 require 改为 import** - 替换所有模块导入语句

5. **将 module.exports 改为 export/export default** - 替换所有模块导出语句

6. **使用打包工具或 babel 处理不兼容情况** - 解决兼容性问题

最终实现项目全面 ESM 化。

## 【问题】
箭头函数与普通函数的区别？

## 【回答】
1. 语法更加简洁、清晰。

2. 箭头函数的 this 默认指向定义它时，所处的上下文的对象的 this 指向，并且即使是 call、apply、bind 等方法也不能改变箭头函数 this 指向。

3. 如果箭头函数是全局函数（外层没有函数包裹）
   a. 在没有使用严格模式的情况下，它的 this 指向的是 window。
   b. 如果开启了严格模式，它的 this 指向 undefined。

4. 箭头函数是没有显示原型 prototype 的，所以不能作为构造函数，也不能使用 new 来创建对象。
   a. 因为在 new 的过程当中，构造函数的 prototype 属性需要赋值给 new 出来的实例对象的 _proto_ 属性，但是箭头函数没有 prototype 属性，所以没有办法作为构造函数。

5. 箭头函数没有自己的 arguments，如果存在函数嵌套，箭头函数会沿着作用域链向上获取外层函数的 arguments 的值。

## 【问题】
?? 和 || 的区别是什么？

## 【回答】
**?? (空值合并运算符):** 只有左值等于 `undefined` 和 `null` 才会返回右值

**|| (逻辑或运算符):** 只有左值为 `false` 时才会返回右值（包括 `false`、`0`、`''`、`null`、`undefined`、`NaN`）

**示例:**
```javascript
0 ?? 'default'  // 0 (0 不是 null/undefined)
0 || 'default'  // 'default' (0 是 falsy)

'' ?? 'default' // '' (空字符串不是 null/undefined)
'' || 'default' // 'default' (空字符串是 falsy)

null ?? 'default' // 'default'
null || 'default' // 'default'
```

## 【问题】
打包器对 CommonJS 和 ESM 的处理方式有何不同？（Webpack、Vite、Rollup）

## 【回答】

### Webpack
- 同时支持 CJS/ESM
- 对 ESM 优化更充分（Tree Shaking）

### Vite / Rollup
- 原生依赖 ESModule
- 优化能力更强
- 对 CJS 会进行转译

现代生态明显偏好 ESM。

## 【问题】
ESM 和 UMD 的区别

## 【回答】

### 1. 本质定位
- **ESM**：ES6 官方推出的**原生模块化标准**，是 JS 语言层面的规范。
- **UMD**：社区通用兼容方案，**非官方标准**，用于同时兼容 CommonJS、AMD 和浏览器全局环境。

### 2. 语法差异
- **ESM**：使用 `import` 导入、`export` 导出，语法简洁规范。
- **UMD**：通过自执行函数判断环境，无统一语法，写法偏兼容适配。

### 3. 加载与优化
- **ESM**：静态加载，编译时解析依赖，**支持 Tree Shaking**，可减小打包体积。
- **UMD**：运行时判断环境，动态适配，**无法 Tree Shaking**。

### 4. 环境支持
- **ESM**：现代浏览器、新版 Node.js 原生支持，是前端主流模块化方案。
- **UMD**：兼容所有环境，包括老旧浏览器、旧版 Node、RequireJS 等。

### 5. 适用场景
- **ESM**：现代项目开发、库开发、追求打包优化。
- **UMD**：库需要提供 CDN 直接引用、兼容老旧环境时使用。

### 总结
ESM 是**现代标准模块化方案**，规范且可优化；UMD 是**通用兼容方案**，主打多环境适配，现在新项目一般优先使用 ESM。

## 【问题】
说出以下程序的运行结果：

```javascript
let x, { x: y = 1 } = { x }; y;
```

## 【回答】

结果为 `1`。

### 执行过程分析：

```javascript
let x, { x: y = 1 } = { x }; y;
```

**步骤拆解：**

1. **`let x`** - 声明变量 `x`，此时 `x` 的值为 `undefined`

2. **`{ x: y = 1 } = { x }`** - 解构赋值：
   - 右侧 `{ x }` 等价于 `{ x: x }`，即 `{ x: undefined }`（因为此时 `x` 还未被赋值，值为 `undefined`）
   - 左侧 `{ x: y = 1 }` 表示：取对象中属性名为 `x` 的值，赋给变量 `y`，如果该值为 `undefined`，则使用默认值 `1`
   - 由于 `{ x }` 中的 `x` 值为 `undefined`，所以 `y` 被赋值为默认值 `1`

3. **`y`** - 返回 `y` 的值，即 `1`

### 关键点：

- 解构赋值中的默认值只在属性值为 `undefined` 时生效
- `let x, { x }` 中的两个 `x` 是不同的：`let x` 声明的是变量，`{ x }` 中的 `x` 是属性名
- 右侧 `{ x }` 是对象简写语法，等价于 `{ x: x }`，此时 `x` 的值为 `undefined`
