# ESM 静态结构

## 2. 为什么 ESM 是静态结构？

### 核心一句话总结

ESM 的 `import`/`export` 语法是**语法级的静态声明**，只能出现在模块顶层，不支持条件判断、变量拼接等动态逻辑，因此打包工具可以在 **`代码执行前（编译阶段）`** 就完整分析模块依赖、导出内容，实现静态优化。

### 详细拆解（面试答）

1. **语法限制**：ESM 强制要求 `import`/`export` 必须在模块**顶层作用域**，不能出现在 `if`/`for` 等条件/循环语句中，也不能用变量作为模块路径（如 `import(moduleName)` 是非法的，只能用 `import()` 动态函数）。

2. **编译时分析**：打包工具（Rollup/Webpack）在编译阶段就能扫描所有 `import`/`export`，构建完整的依赖图，确定每个模块的导出内容、哪些被使用、哪些未被使用。

3. **静态优化的基础**：正是这种静态结构，才让 Tree-shaking、作用域提升、代码分割等优化成为可能，而 CommonJS 是运行时动态加载，无法在编译阶段做这些分析。

### 补充：动态 `import()` 是例外吗？

不是。静态 `import` 是 ESM 的核心，`import()` 是 ESM 为了支持按需加载补充的**函数式动态导入**，返回 Promise，属于运行时加载，不影响 ESM 整体的静态结构。
