# HTTP/2 相比 HTTP/1.1，主要解决了哪些核心问题？

【问题】HTTP/2 相比 HTTP/1.1，主要解决了哪些核心问题？

【回答】主要解决了 HTTP/1.1 在现代高并发网页下的五大性能瓶颈：

1. **队头阻塞（Head-of-Line Blocking）**：一个连接里，一个请求慢会阻塞后续所有请求。

2. **连接数过多**：浏览器对同域名的 TCP 连接数有限制（通常 6-8 个），导致并行请求受限。

3. **头部冗余**：每次请求都重复发送大量头部信息（Cookie、User-Agent 等），浪费带宽。

4. **服务器被动**：服务器无法主动预判客户端需求，只能等待请求。

5. **纯文本协议低效**：解析纯文本协议开销大，效率低。

# HTTP/2 的核心优势是什么？

【问题】HTTP/2 的核心优势是什么？

【回答】HTTP/2 通过底层重构，带来了三大核心优势：

1. **多路复用（Multiplexing）**：这是最核心的优势。在单个 TCP 连接上，通过二进制帧和流 ID，并发处理多个请求和响应，彻底解决了队头阻塞，大幅提升连接利用率。

2. **头部压缩（HPACK）**：使用静态和动态字典算法，对重复的头部进行高效压缩，显著减少传输体积。

3. **服务器推送（Server Push）**：服务器可以主动向客户端推送其即将需要的资源，无需等待客户端解析 HTML 后再发起请求，减少网络往返时间（RTT）。

# 请详细解释一下 HTTP/2 的"服务器推送"（Server Push）功能

【问题】请详细解释一下 HTTP/2 的"服务器推送"（Server Push）功能。

【回答】服务器推送是指服务器在收到客户端对一个资源（如 HTML）的请求后，**主动预判并推送**客户端后续肯定需要的关键资源（如 CSS、JS、字体）。

- **原理**：服务器通过 PUSH_PROMISE 帧预先告知客户端将要推送的资源，并直接发送数据。

- **好处**：消除了客户端解析 HTML 再发起请求的过程，显著减少了 RTT 往返，从而加速首屏加载。

- **实践场景**：广泛用于电商首页、博客详情页、SPA 应用等场景，专门推送核心 CSS、关键 JS 和字体文件。

# 在实际项目中，如何配置和使用 HTTP/2 的服务器推送？

【问题】在实际项目中，如何配置和使用 HTTP/2 的服务器推送？

【回答】主流服务器和 CDN 均支持，配置方式很简单，以 Nginx 为例：

1. **前提**：监听端口需开启 http2 模式。

2. **开启配置**：使用 `http2_push_preload on;` 指令。

3. **指定资源**：通过 Link 响应头，配合 `rel=preload` 告知要推送的资源。

```nginx
server {
    listen 443 ssl http2;
    # 开启推送
    http2_push_preload on;
    
    location ~ \.html$ {
        # 推送核心 CSS 和 JS
        add_header Link "</css/main.css>; rel=preload; as=style";
        add_header Link "</js/app.js>; rel=preload; as=script";
    }
}
```

- **注意**：实际使用时**切勿滥用**，只推送体积小、优先级高、必加载的资源（3-5 个以内），并配合缓存策略避免重复推送。

# HTTP/2 相比 HTTP/1.1，在性能上具体提升了多少？

【问题】HTTP/2 相比 HTTP/1.1，在性能上具体提升了多少？

【回答】根据 Google 和 Akamai 的实测数据，在典型的 Web 场景下：

- **首屏加载时间（First Contentful Paint）** 平均可提升 **10% - 30%**。

- **在高延迟网络（如 3G/4G 移动网络）下**，性能提升更为明显，部分场景可达 **50% 以上**。

- **核心原因**是多路复用消除了队头阻塞，并减少了 TCP/TLS 握手次数。

# HTTP/1.1 和 2.0 的区别是什么？

HTTP/2 相比 HTTP/1.1 性能上的改进：

- **头部压缩**：HTTP/2 会压缩头（Header）如果你同时发出多个请求，他们的头是一样的或是相似的，那么，协议会帮你消除重复的部分。这就是所谓的 HPACK 算法：在客户端和服务器同时维护一张头信息表，所有字段都会存入这个表，生成一个索引号，以后就不发送同样字段了，只发送索引号，这样就提高速度了。

- **二进制格式**：HTTP/2 不再像 HTTP/1.1 里的纯文本形式的报文，而是全面采用了二进制格式，头信息和数据体都是二进制，并且统称为帧（frame）：头信息帧（Headers Frame）和数据帧（Data Frame）。这样虽然对人不友好，但是对计算机非常友好，因为计算机只懂二进制，那么收到报文后，无需再将明文的报文转成二进制，而是直接解析二进制报文，这增加了数据传输的效率。

- **并发传输**：引出了 Stream 概念，多个 Stream 复用在一条 TCP 连接。解决了 HTTP/1.1 队头阻塞的问题：

- **服务器主动推送资源**：HTTP/2 还在一定程度上改善了传统的「请求 - 应答」工作模式，服务端不再是被动地响应，可以主动向客户端发送消息。
