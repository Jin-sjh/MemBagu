# 前端死锁问题详解

【问题】
什么是前端的死锁？前端实际开发中会遇到哪些死锁情况？如何在开发中避免死锁？

【回答】

## 1. 核心概念

**面试官问：什么是前端的死锁？**

前端的死锁本质上是**异步任务或资源之间的相互等待**。在 JS 单线程模型下，通常表现为 Event Loop 中的任务队列出现了 "循环依赖" 或 "无限阻塞"，导致主线程被卡死，后续代码无法执行，页面完全无响应。它通常需要满足四个经典条件：互斥、占有并等待、不可抢占、循环等待。

## 2. 常见场景（5 大高发点）

**面试官问：前端实际开发中会遇到哪些死锁情况？**

前端死锁主要集中在以下 5 个场景：

1. **Promise 永远不 resolve**：最常见。例如写了 `await new Promise(() => {})`，导致后续 async 逻辑永久阻塞，Event Loop 空转。

2. **WebWorker 循环等待**：Worker A 等 Worker B 的消息，同时 Worker B 又等 Worker A 的消息，双方互相等待导致死锁。

3. **Service Worker 循环请求**：页面等待 SW 返回结果，SW 又同步等待页面某个事件，双方互相等待导致卡死。

4. **SharedArrayBuffer + Atomics 阻塞**：使用共享内存时，某一线程持锁不释放，另一线程调用 `Atomics.wait()` 永久阻塞。

5. **同步 XHR + 资源竞争**：主线程执行同步 XHR 被阻塞，但 XHR 的事件处理又需要主线程来执行，导致系统假死。

## 3. 避免策略（背诵口诀）

**面试官问：如何在开发中避免死锁？**

避免死锁可以遵循以下 6 点原则：

1. **杜绝 "无限等待"**：避免写永不 resolve/reject 的 Promise，必要时手动加超时处理。

2. **规范 Worker 通信**：Worker 间通信必须设置超时机制，避免双向无限等待。

3. **慎用共享资源**：尽量减少共享可变状态，多使用不可变数据结构（Immutable）从根源解决竞争问题。

4. **规避同步 API**：坚决避免使用同步 XHR，统一使用异步 Fetch 或 Axios。

5. **合理管理 SW 生命周期**：Service Worker 与页面间的通信要设计清晰的状态机，避免循环依赖。

6. **原子操作规范化**：使用 Atomics 时严格遵循释放规则，避免线程长期占锁。

## 💡 10 秒速记口诀

为了方便你在现场记忆，可以记这个口诀：

**"防永等，设超时；少共享，弃同步；SW 规范，Atomics 放。"**
