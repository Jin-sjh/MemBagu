# 内存泄漏风险与优化

## 【问题】
垃圾回收 (GC) 的核心机制是什么？

## 【回答】
JavaScript 采用自动垃圾回收机制，主流浏览器使用标记 - 清除算法：

1. **标记阶段**：从根对象（window、全局执行上下文）出发，遍历所有可达对象，标记为「存活」

2. **清除阶段**：回收所有未被标记的「不可达」对象，释放内存

3. **内存泄漏本质**：对象不再被使用，但因强引用存在，被 GC 误判为「可达」，无法被回收

## 【问题】
哪些操作会造成强引用？

## 【回答】
强引用是指默认的引用关系，只要强引用存在，垃圾回收器（GC）永远不会回收被引用的对象，是内存泄漏的核心诱因。常见场景：

- **全局变量 / 挂载到 window**：`window.obj = {}`，全局对象生命周期与页面一致，无法被回收

- **未销毁的定时器 / 延时器**：`setInterval`、`setTimeout` 回调中持有 DOM / 组件引用，定时器不清除则引用不释放

- **事件监听未解绑**：`addEventListener` 绑定后未用 `removeEventListener` 移除，DOM 被销毁但事件回调仍持有引用

- **闭包持有外部变量**：闭包会保留对外部作用域变量的引用，导致变量无法被回收

- **缓存 / 长列表未清理**：无限滚动列表中，所有数据都存在内存中不做清理

- **第三方库实例未销毁**：如 ECharts、地图实例未调用 `dispose()`，持续持有 DOM 引用

- **Vue/React 组件内不当引用**：Vue 中 `this` 的循环引用、React 中未清理的 `useEffect` 副作用

## 【问题】
内存泄漏的常见泄漏场景有哪些？

## 【回答】
1. **意外的全局变量**：
   ```javascript
   function foo() { bar = 1; }
   ```
   未声明直接赋值会创建全局变量，导致无法被垃圾回收

2. **闭包过度使用**：长期缓存大对象，闭包让外层作用域的变量无法被垃圾回收

3. **DOM 事件未移除**：组件卸载后事件监听仍然存在，导致 DOM 元素无法被回收

## 【问题】
这段代码是否存在内存风险？为什么？

## 【回答】
这段代码本身不存在内存泄漏风险。

1. 代码是典型的闭包实现，内部函数持有外部函数 `createCounter` 的词法环境（变量 `count`）。

2. 闭包本身不会造成内存泄漏，内存泄漏的核心是「不再需要的内存无法被 GC 回收」。

3. 当闭包函数不再被任何变量引用时，其持有的词法环境会被正常回收；只有当闭包被意外长期持有（如全局挂载、未清理的定时器 / 事件监听）时，才会引发内存泄漏。

4. 闭包是前端工程的必需品，用于状态封装、模块化等场景，不能简单认为"闭包会造成内存泄漏"。

## 【问题】
如何避免长列表内存堆积？

## 【回答】
核心是减少 DOM 节点和及时释放引用：

1. **虚拟列表**：只渲染可视区域元素，复用 DOM

2. **分页 / 懒加载**：避免一次性加载全部数据

3. **组件卸载时销毁数据和事件**：及时清理无用引用

4. **用弱引用存储缓存**：使用 WeakMap/WeakSet，不影响 GC

5. **避免全局缓存**：在组件作用域内管理数据

## 【问题】
内存泄漏的优化方案有哪些？

## 【回答】
1. **及时解除引用**：
   ```javascript
   timer = null;
   ```
   将不再使用的变量引用置为 null，帮助垃圾回收器识别

2. **使用 WeakMap 存储临时数据**：WeakMap 的键是弱引用，不会阻止垃圾回收

3. **组件卸载时清理事件监听**：在组件销毁前主动移除所有事件监听器，避免内存泄漏
