【问题】前端性能优化主要从哪些方面入手？（简洁版，1 分钟内说完）

【回答】
前端性能优化主要从网络、资源加载、渲染、代码、运行时几个方面入手：
1. 网络优化：减少请求数，合并资源，开启 Gzip/Brotli 压缩，使用 CDN，设置合理缓存。
2. 资源加载优化：图片懒加载、按需加载路由、使用 webp/avif 等现代图片格式、压缩图片。
3. 渲染优化：减少重排重绘，使用 CSS3 硬件加速，避免频繁操作 DOM。
4. 代码优化：减少冗余代码，Tree Shaking，避免内存泄漏，防抖节流。
5. 构建优化：代码分割、压缩混淆、第三方库按需引入。

【问题】请详细谈谈前端性能优化的完整方案（面试高分回答，条理清晰）

【回答】
我一般从网络层、资源层、渲染层、代码层、构建层、运行时六个维度做性能优化：

1. 网络层面
- 启用 HTTP/2、HTTP/3，多路复用减少请求阻塞
- 开启 Gzip / Brotli 压缩文本资源
- 使用 CDN 加速静态资源
- 配置强缓存与协商缓存（Cache-Control、ETag、Last-Modified）
- 减少 DNS 解析，使用 DNS Prefetch、Preconnect

2. 资源加载优化
- 图片优化：压缩、WebP/AVIF、懒加载、响应式图片 srcset
- 字体优化：font-display: swap，避免 FOIT/FOUT
- 脚本优化：script defer/async，关键 JS 内联
- 路由懒加载、组件异步加载，实现按需加载

3. 渲染优化（浏览器渲染流程）
- 减少 重排（Reflow）重绘（Repaint），主要从以下几方面做：
  - 读写分离：避免在循环或高频事件里读写 DOM 穿插，防止强制同步布局
  - 批量操作 DOM：使用 DocumentFragment 离线修改，只做一次插入
  - 动画优先用 transform 和 opacity：让浏览器生成独立合成层，用 GPU 加速，不触发重排重绘
  - 把频繁变化的元素脱离文档流：缩小重排影响范围
  - 避免频繁读取布局属性：减少 Layout 次数，提升渲染效率
- 批量 DOM 操作，使用文档片段 DocumentFragment
- 避免频繁读取 offsetTop/scrollTop 等布局属性
- 使用 transform、opacity 触发合成层，硬件加速
- 避免大面积使用 * 通配符、昂贵选择器

4. JavaScript 运行时优化
- 防抖（debounce）、节流（throttle）处理高频事件
- 避免内存泄漏（定时器未清除、全局变量滥用、闭包过度）
- 长任务拆分，使用 Web Worker 处理计算密集型逻辑
- 避免在 render / 更新中做复杂计算

5. 构建打包优化
- Tree Shaking 删除无用代码
- 代码分割（Code Splitting）
- 第三方库按需引入，替换轻量替代库
- 压缩 JS/CSS/HTML，移除 console、注释
- 分析打包体积（webpack-bundle-analyzer）

6. 体验与监控
- 使用骨架屏、loading 提升感知性能
- 预加载关键资源：preload、prefetch
- 接入 Lighthouse、Web Vitals 监控 FCP、LCP、CLS、FID 等指标

b. 资源压缩与合并：压缩 JavaScript、CSS 和图片等资源，减少文件大小和请求次数。
c. 代码分割：使用 Webpack 等工具实现按需加载，减少初始加载的资源大小。

【回答】
a. 减少 HTTP 请求数：合并文件、使用图标字体等。
b. CDN 加速：使用 CDN 缓存静态资源，提高加载速度。
c. 预加载：使用 `<link rel="preload">` 预加载关键资源。

【问题】通用工程化优化技巧有哪些？（Vue/React 通用）

【回答】
1. 用 Vite 替代 Webpack（开发/生产）：Vite 基于 Rollup，原生支持 Tree-shaking，打包速度更快，体积更小。

2. 配置 sideEffects 字段：在 package.json 中标记无副作用的文件，让打包工具安全删除死代码：
```json
{
  "sideEffects": [
    "*.css",
    "src/global.js",
    "src/polyfill.js"
  ]
}
```

3. 代码分割（Code Splitting）：除了路由懒加载，还可以按第三方库分割（如把 vue/react、element-plus 打包成独立 chunk，利用浏览器缓存）。

4. 压缩优化：配合 Terser 压缩 JS、PostCSS 压缩 CSS、图片压缩（vite-plugin-imagemin），进一步减小体积。

【问题】请求缓存的实现方式与核心优势？

【回答】
原理：对相同参数的接口请求，将结果缓存到内存 / 本地，重复请求直接返回缓存，减少网络请求

实现方式：
- 内存缓存：Map / WeakMap 存储请求结果，设置过期时间
- 本地缓存：localStorage / sessionStorage / IndexedDB 存储静态数据
- 框架层面：Axios 拦截器、React Query、SWR 等工具封装

核心优势：减少服务器压力，提升页面响应速度，避免重复请求导致的性能问题

【问题】虚拟列表（Virtual Scrolling）的实现原理是什么？

【回答】
原理：计算可视区域高度、单个 item 高度，只渲染可视区域内的 N 个元素，滚动时动态更新渲染列表，复用 DOM 节点

核心优势：将长列表的 DOM 节点数从万级降到百级，大幅减少重排重绘，解决内存堆积

面试考点：实现原理、滚动计算、占位容器、动态高度处理、性能优化

适用场景：无限滚动列表、大数据表格、聊天记录等

5. 体积分析工具：用 rollup-plugin-visualizer（Vite）、webpack-bundle-analyzer（Webpack）分析打包体积，定位大文件，针对性优化。

【问题】请谈谈你如何系统性优化前端性能？（四阶段答题模板）

【回答】
前端性能优化是一个系统工程，我将按照页面加载的生命周期，从四个维度来进行优化：

**第一阶段：网络请求层 —— 尽量少快好**
- 减少请求数：合并 CSS/JS 文件；图片使用 WebP 格式并按需懒加载（loading="lazy" 或 IntersectionObserver）；第三方库按需引入（如 Lodash 用 ES 版本）。
- 加快请求速度：静态资源上 CDN；使用 HTTP/2 多路复用；通过 DNS 预解析（dns-prefetch）和预连接（preconnect）降低延迟。

**第二阶段：资源构建层 —— 拆分与缓存**
- 代码分割：利用 Webpack/Vite 的 SplitChunks 和动态 import()，实现路由级和组件级的按需加载。
- Tree Shaking：利用 ES6 模块特性，删除未使用的死代码。
- 长久缓存：文件名加 Hash，配合 Cache-Control: max-age 设置强缓存，代码更新后 Hash 变化自动失效。

**第三阶段：浏览器渲染层 —— 避免卡顿和白屏**
- 关键渲染路径：CSS 放 head，JS 放 body 底部或加 defer/async，防止阻塞 DOM 解析。
- 减少重排重绘：操作 DOM 前先脱离文档流（合并样式改用 class）；动画尽量使用 transform 和 opacity 触发 GPU 硬件加速。
- 大数据渲染：对于长列表使用虚拟列表，只渲染可视区域内的 DOM。

**第四阶段：JS 执行层 —— 释放主线程**
- 防抖节流：对 scroll、resize、input 等高频事件进行控制。
- 计算密集型任务：使用 Web Worker 将复杂计算放到后台线程，避免阻塞 UI 导致掉帧；或者使用 requestIdleCallback 进行任务切片。
- 避免内存泄漏：及时清除定时器、解除事件绑定，注意闭包对大对象的引用。

**【总结收尾】**
最后，我认为性能优化不能盲目，一定要以数据为导向。我通常会结合 Lighthouse 和 Chrome Performance 面板先定位瓶颈，特别是关注 Core Web Vitals 指标（LCP、INP、CLS），找到最拖后腿的一环，再对症下药，避免过度优化增加维护成本。

【问题】你是如何思考和衡量性能优化的？（核心指标答题模板）

【回答】
我现在的优化思路主要是按照 Google 提出的 Core Web Vitals（核心网页指标）来展开的，因为它们直接关联到用户的真实体验和 SEO：

**1. 针对 LCP（最大内容渲染时间 - 衡量加载速度）**
"如果 LCP 慢，说明首屏加载有问题。我会排查是否是静态资源太大、服务端响应太慢。优化手段包括：首屏关键资源使用 `<link rel="preload">` 预加载、图片转 WebP 并响应式图片（srcset）、优化关键渲染路径（CSS 提前、JS 异步）。"

**2. 针对 INP（交互到下次响应 - 衡量响应速度）**
"如果 INP 慢，说明主线程卡住了。我会排查是否有长任务。优化手段包括：把复杂的计算逻辑下沉到 Web Worker；对长任务进行时间切片，把控制权交还给浏览器渲染；对高频交互事件严格使用防抖节流。"

**3. 针对 CLS（累积布局偏移 - 衡量视觉稳定性）**
"如果 CLS 高，说明页面会跳动。这通常是图片/动态内容没有占位导致的。优化手段包括：给图片和视频显式设置 width 和 height；避免在用户已可视的区域上方动态插入 DOM；使用骨架屏代替传统的 Loading 闪屏。"

【问题】如何监测和定位性能问题？

【回答】
1. **被动监测到慢到快**：核心是监听长任务，计算 startIndex 和 endIndex，只渲染这几条数据，并通过 transforms translateY 制造滚动假象，配合缓冲区（上下多渲染几条）防止快速滚动时白屏。

2. **被问到强缓存和协商缓存的区别**：强缓存不发送请求（状态码 200 from disk/memory），靠 Cache-Control 或 Expires；协商缓存要发请求验证，未改变返回 304，靠 ETag/If-None-Match 或 Last-Modified/If-Modified-Since。

3. **展示你踩过坑**："之前项目遇到首屏过慢，我用 Performance 面板一看，发现是一个很大的第三方库全量引入了，直接按需引入并做了 Tree Shaking 后，打包体积直接减少了 40%。"

【问题】CSS Containment + will-change 的核心作用是什么？

【回答】
**contain: paint** - 限制重绘范围，确保元素变化只影响自身，不波及父元素或兄弟元素

**will-change** - 提前告知浏览器元素即将变化，让浏览器：
- 创建独立图层
- 启用 GPU 硬件加速
- 预分配资源优化渲染

【问题】拖拽场景的优化方案有哪些？

【回答】
1. **使用 transform** - 避免重排重绘，利用 GPU 加速
2. **配合 will-change** - 提示浏览器提前优化
3. **使用 requestAnimationFrame** - 控制渲染时机，避免高频渲染

【问题】前端性能优化的要点有哪些？

【回答】
- transform 不会触发重排，性能优于 top/left
- will-change 需谨慎使用，只在确实会变化的元素上设置
- RAF 确保在浏览器合适的渲染周期内更新

【问题】requestAnimationFrame 优化拖拽渲染的原理是什么？

【回答】requestAnimationFrame 会在浏览器下一次重绘之前调用指定的函数。在拖拽场景中，它能让拖拽相关的操作（如更新元素位置等）与浏览器的刷新频率同步。这样可以避免不必要的重绘和回流，提高渲染效率，使得拖拽过程更加流畅，减少卡顿现象。因为它是在浏览器准备好进行下一次渲染时才执行，能更好地利用系统资源，以合适的时机去更新界面状态，从而优化了拖拽过程中的渲染表现。

【问题】INP 是什么？怎么统计的？

【回答】INP（Input Delay）即输入延迟，它反映了从用户输入到浏览器做出响应的时间。统计通常是通过浏览器的性能监控机制，在用户进行输入操作（如点击、键盘输入等）时开始计时，到浏览器实际处理并做出响应时结束计时，这个时间差就是 INP。

【问题】INP 具体是怎么做的优化？

【回答】优化 INP 可以从多个方面着手。比如优化页面的渲染性能，减少重绘和回流，使页面加载和交互更加流畅。优化脚本的执行效率，避免长时间运行的脚本阻塞主线程。还可以采用一些技术，如惰性加载，只在需要时加载相关资源，减少初始加载时间，从而提高整体的响应速度，降低 INP。
