# 6. 死锁的面试题（前端能遇到什么类型？）（必会）

【问题】
前端能遇到什么类型的死锁？如何避免？

【回答】

## (1) 什么是死锁？

多个线程/任务彼此等待对方释放资源，最终所有人都无法继续。

经典条件（同时满足才会死锁）：

1. 互斥（资源只能一个线程使用）
2. 占有并等待
3. 不可抢占
4. 循环等待

## (2) 前端实际中可能出现的死锁场景

### 场景 1：WebWorker 间消息循环导致互相等

- Worker A 等 B 回消息
- Worker B 又等 A 回消息
- 双方都卡死

### 场景 2：SharedArrayBuffer + Atomics

JS 中唯一可能 "锁住" 的情况
例如线程 A 持有锁不释放，线程 B 一直阻塞 Atomics.wait

### 场景 3：Service Worker 与页面之间循环请求

- 页面等待 SW 返回
- SW 又同步等待页面某个事件
- → 卡死

### 场景 4：Promise 死锁（任务永远不 resolve）

最常见于编码错误：

```javascript
await new Promise(() => {}) // 永不 resolve
```

导致：
- 后续 async 逻辑全部卡住
- EventLoop 空转

### 场景 5：同步 XHR + 主线程资源竞争

- 主线程执行同步 XHR
- XHR 需要主线程处理事件
- 双方等待 → 页面完全假死

## (3) 如何回答如何避免死锁？

可以直接背这个：

1. 尽量避免共享可变状态
2. 使用不可变数据结构
3. 避免长时间持有资源
4. Worker 间通信要有超时机制
5. 避免使用同步 XHR
6. 避免与永不 resolve 的 Promise
