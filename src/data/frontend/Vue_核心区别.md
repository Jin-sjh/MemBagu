# Vue 核心区别

## 【问题】
为什么 Vue 组件 data 必须用函数返回对象？

## 【回答】
在 Vue 中，组件的 data 必须是一个函数，并且返回一个对象。

因为组件是可以复用的，而 JavaScript 里对象是引用类型。

如果 data 直接写成对象，多个组件实例会共用同一个数据对象，一个实例改数据，所有实例都会受影响，造成数据污染。

用函数返回对象，每创建一个实例就会执行一次函数，生成独立的数据副本，从而保证每个实例的数据相互隔离、互不干扰。

## 【问题】
Vue MVVM 架构模式是什么？

## 【回答】
Vue 采用 MVVM（Model-View-ViewModel）架构模式。
Model 是数据层，在 Vue 中通常是 data、ref、reactive 等响应式数据；
View 是视图层，对应 template 模板生成的 DOM；
ViewModel 是 Vue 实例本身，负责连接 Model 和 View。

Vue 通过响应式系统（Vue2 使用 Object.defineProperty，Vue3 使用 Proxy）对数据进行代理和依赖收集，当数据变化时会触发 Watcher / effect 重新执行渲染函数，通过 Virtual DOM diff 更新真实 DOM，从而实现数据驱动视图更新。同时通过事件绑定或 v-model 实现视图到数据的更新。

## 【问题】
Object.defineProperty 和 Proxy 本质区别？

## 【回答】
两者最核心的区别是**拦截粒度不同**。

- **Object.defineProperty** 是**属性级别**的劫持，只能劫持对象**已存在的单个属性**，无法监听新增 / 删除属性，也不能原生监听数组下标和 length。

- **Proxy** 是**对象级别**的代理，直接代理整个对象，能监听对象的**新增、删除、修改、数组下标、length** 等所有操作，支持 13 种拦截方式，功能更强大。

简单说：**一个拦属性，一个拦对象**。

## 【问题】
Vue2 和 Vue3 的核心区别是什么？

## 【回答】
Vue2 和 Vue3 的主要区别有四点：
第一，响应式系统从 Object.defineProperty 升级为 Proxy，可以监听对象新增、删除和数组索引变化，性能更好。
第二，API 风格从 Options API 变成 Composition API，通过 setup 组织逻辑，使代码复用性和可维护性更强。
第三，Vue3 在性能上做了很多优化，比如 Fragment、静态提升、Patch Flag、Teleport、Suspense 等。
第四，Vue3 支持 Tree-shaking 和 TypeScript，体积更小，更适合大型工程开发。

## 【问题】
Vue 生命周期是什么？

## 【回答】

Vue 的生命周期，我通常会从它的四个核心阶段来理解和使用，分别是：创建、挂载、更新和销毁。

首先是创建阶段，主要涉及 beforeCreate 和 created 钩子。执行到 created 时，组件的响应式数据（data）和方法（methods）已经初始化完成了，但真实的 DOM 还没有生成。所以我通常会在 created 里做一些基础变量的初始化，或者发起一些完全不依赖 DOM 结构的异步请求，让拉取数据的动作尽早发生。

其次是挂载阶段，主要涉及 beforeMount 和 mounted。当虚拟 DOM 转换成真实 DOM 并挂载到页面后，就会触发 mounted，这也是我平时用得最多的钩子。只要涉及到 DOM 操作，比如需要获取某个元素的宽高，或者需要初始化像 ECharts、Swiper 这样的第三方库，我都一定会放在 mounted 里执行。

另外，关于发请求的时机，虽然在 created 发送能早一点点，但在实际复杂业务中，如果请求回来的数据需要结合 DOM 进行后续计算，或者项目涉及服务端渲染（SSR，因为 mounted 不会在服务端执行），为了统一管理和避免意外的报错，我也经常将核心的业务请求放在 mounted 中处理。

接下来是更新阶段，也就是 beforeUpdate 和 updated。当响应式数据发生改变，导致页面重新渲染完毕后会触发 updated。这里在开发中有一个必须要避开的坑：就是绝对不能在 updated 钩子里再去修改相关的响应式数据，否则会触发再次更新，直接导致无限死循环。

最后是销毁阶段，即 beforeUnmount 和 unmounted（在 Vue 2 中叫 destroyed）。这是组件卸载前的"收尾大扫除"阶段。为了防止内存泄漏，我一定会在这里集中清理副作用。具体来说，凡是我在前面开启的 setInterval 定时器、绑定的 window.addEventListener 全局滚动或缩放事件，我都会在这里用对应的 clear 或 remove 方法把它们彻底解绑和清除干净。

最后补充一点，目前我主要使用 Vue 3 的组合式 API，在 <script setup> 语法糖下，原本属于 created 阶段的逻辑我直接写在 setup 顶层，其他的生命周期则通过引入 onMounted、onUnmounted 等函数来使用，这让我的代码逻辑拆分和复用变得更加方便了。

## 【问题】
Vue 3 的响应式原理是怎么实现的？（核心概念）

## 【回答】
Vue 3 底层是通过 ES6 的 Proxy 对象对数据（对象或数组）进行代理，拦截对其属性的读取（get）和设置（set）操作。

- 当组件渲染或 watcher 执行时，如果"访问"了某个数据属性，Vue 会触发 get 拦截，并收集依赖（track）。

- 当该数据被修改时，Vue 会触发 set 拦截，并触发依赖更新（trigger），从而通知对应的组件或 watcher 重新渲染。

## 【问题】
ref 和 reactive 有什么区别？

## 【回答】
主要在支持的数据类型、底层响应方式和使用场景上有所区别：

- 数据类型：reactive 仅支持对象和数组（引用类型）；而 ref 既支持基本数据类型（如数字、字符串），也支持对象和数组。

- 响应方式：reactive 直接通过 Proxy 拦截对象的属性；ref 则是将其包装成一个带有 value 属性的对象，通过拦截 .value 来实现响应式。

- 使用场景：reactive 适合处理包含多个属性的复杂对象或嵌套对象；ref 更适合单值、简单的状态，在组合式 API 中使用最为广泛。

## 【问题】
请详细解释一下依赖收集（track）和触发更新（trigger）的过程？

## 【回答】

- 依赖收集（track）：当代码读取 reactive 或 ref 的属性时，会触发代理的 get。此时，Vue 会把当前正在运行的 effect（比如组件的渲染函数、computed 或 watcher）记录下来，并放进这个属性的依赖集合里。简单来说，就是记录"谁用了这个数据"。

- 触发更新（trigger）：当这个属性的值被修改时，会触发代理的 set。Vue 会去查找这个属性对应的依赖集合，把之前记录的 effect 全部找出来并重新执行，从而驱动视图更新。简单来说，就是"数据变了，通知用过它的人重新干活"。

## 【问题】
Vue3 是如何处理深层对象和数组的？

## 【回答】

- 深层嵌套对象：Vue 3 采用的是"懒代理"（惰性代理）。Proxy 只有在对象属性被真正访问到那层时，才会按需进行递归代理和依赖收集，这就大大提升了初始化的性能。

- 数组处理：Proxy 原生就能拦截数组索引和长度的变化。此外，Vue 内部还拦截了数组的变异方法（如 push、pop、splice 等），确保对数组的增删改操作也能正确触发视图的更新。

## 【问题】
Vue3 改用 Proxy 解决了哪些痛点？

## 【回答】
Vue3 使用 Proxy 主要解决了 Vue2 响应式的几大痛点：

1. **支持对象新增 / 删除属性**，不再需要 $set / $delete；

2. **支持数组下标修改和 length 修改**，无需重写数组方法；

3. **实现懒递归（惰性监听）**，用到哪一层才代理哪一层，初始化性能更高、内存占用更小；

4. **不污染原对象**，返回新的代理对象，更安全优雅；

5. **支持更多操作拦截**，如 in、delete、遍历等，扩展性更强。

## 【问题】
一句话总结 Vue 3 的响应式原理？（面试话术总结）

## 【回答】
"Vue 3 的响应式原理主要是通过 Proxy 对对象或数组进行代理拦截。在读取数据时通过 track 收集依赖，在修改数据时通过 trigger 触发更新。它提供了 ref 用于基本类型（通过 .value 拦截），以及 reactive 用于对象或数组，实现了数据变化自动驱动视图更新，并高效支持了深层属性的按需代理。"

## 【问题】
虚拟 DOM 和 Diff 算法的作用？

## 【回答】
Vue 通过虚拟 DOM 将真实 DOM 抽象为 JS 对象树，当数据变化时会生成新的虚拟 DOM，然后通过 Diff 算法对比新旧虚拟 DOM，只更新变化的节点，从而减少真实 DOM 操作，提高渲染性能。

在列表渲染中使用 key 可以唯一标识节点，使 Diff 算法能够复用已有节点，避免不必要的 DOM 重建，同时还能保证组件状态稳定，例如输入框内容不会错乱。

## 【问题】
为什么虚拟 DOM Diff 要同层比较，不跨层递归？

## 【回答】
传统树 diff 是全量递归比较，时间复杂度是 O(n³)，节点一多性能会非常差。

而实际业务里，DOM 节点跨层级移动的场景极少，大部分都是同层更新。

所以 React、Vue 都采用同层比较，只对比同一层级的节点，一旦发现节点层级变了，就直接销毁重建，这样把时间复杂度降到 O(n)，在性能和实用性之间做了最优权衡。

## 【问题】
Key 在 Diff 过程中起什么作用？

## 【回答】
Key 是虚拟 DOM 中节点的唯一标识。

Diff 算法在做同层比较时，会通过 key 快速判断新旧节点是否为同一个。

尤其是列表渲染时，有 key 可以精准复用 DOM 节点，只做移动，不重复创建销毁。

没有 key 或用错 key（比如用索引），会导致节点复用错误、渲染异常，还会影响性能。

**要点**：
1. 唯一标识，判断是否是同一个节点
2. 帮助 Diff 算法复用节点，提升性能
3. 避免列表渲染错乱、状态混乱
4. 不能用随机数/索引（容易踩坑）

## 【问题】
Vue Router 的导航守卫有哪些？分别用于什么场景？

## 【回答】
Vue Router 的导航守卫可以分为三类：全局守卫、路由独享守卫和组件内守卫。

**全局守卫**：
- **beforeEach**：在每次路由跳转开始时最先执行，常用于登录状态校验、权限判断和路由白名单控制。如果不满足条件，可以中断或重定向导航。
- **beforeResolve**：会在所有守卫执行完成、组件渲染之前触发，适合依赖最终路由状态的逻辑，例如精确埋点或确认跳转结果。
- **afterEach**：在路由跳转完成后执行，不能中断导航，一般用于页面访问统计、修改 document.title 等纯副作用操作。

**路由独享守卫**：
- **beforeEnter**：通过在路由配置中使用，只对某一个路由生效，适合某些页面有特殊权限或前置条件判断的场景，避免将逻辑写进全局守卫。

**组件内守卫**：
- **beforeRouteEnter**：进入组件前可以做数据校验或权限确认
- **beforeRouteUpdate**：路由参数变化时触发
- **beforeRouteLeave**：离开组件时常用于表单未保存的离开提示

**面试话术**：
Vue Router 的守卫用于控制路由跳转流程，全局守卫适合通用的权限和登录控制，路由独享守卫适合单页面特殊逻辑，组件内守卫更关注页面级生命周期和用户操作控制。

## 【问题】
Vue 的响应式原理？

## 【回答】
Vue 的响应式核心是**依赖收集**和**派发更新**，底层基于**发布 - 订阅模式**，由 Dep 和 Watcher 协作完成：

**1. 数据劫持**
Vue2 通过 Object.defineProperty 为每个属性设置 getter/setter，Vue3 使用 Proxy 代理，实现对数据读写的拦截。

**2. 依赖收集（getter 中触发）**
当组件渲染或计算属性执行时，会创建对应的 Watcher，并将其标记为**当前激活的 Watcher**。
此时读取响应式数据，会触发 getter，属性对应的 Dep（依赖管理器）就会把当前 Watcher 收集到自己的订阅列表里。
这一步就是**依赖收集**。

**3. 派发更新（setter 中触发）**
当修改响应式数据时，会触发 setter，Dep 会遍历自己的订阅列表，调用所有 Watcher 的 update 方法，通知它们更新。
这一步就是**派发更新**。

**4. Dep 与 Watcher 关系**
- Dep 是**发布者**，管理多个 Watcher；
- Watcher 是**订阅者**，依赖多个 Dep；
- 两者是**多对多**的发布 - 订阅关系。

## 【问题】
Vue 中 computed 和 watch 有什么区别？

## 【回答】
computed 是计算属性，具备缓存和懒加载特性，只有依赖数据改变时才会重新计算，适合多数据计算出一个结果、同步且需要复用的场景；watch 是侦听器，没有缓存，数据变化就会触发，适合监听频繁变动的数据、执行异步操作或处理复杂副作用逻辑，二者一个侧重计算结果，一个侧重监听变化。

## 【问题】
Vue Computed 和 Watch 的核心区别是什么？

## 【回答】
一句话概括：**算状态用 Computed，做操作用 Watch**。

**Computed（计算属性）**：
- 本质是派生状态，基于已有状态计算新值
- 有缓存机制，依赖不变时复用缓存，性能高
- 懒执行，只有被访问时才计算
- 必须同步返回结果，不能做异步操作
- 自动追踪深层依赖
- 典型场景：列表过滤/排序、计算总和、组合状态展示

**Watch（侦听器）**：
- 本质是副作用函数，状态变化后做额外事情
- 无缓存，每次变化都执行
- 数据变化立即执行，可配置 immediate 立即跑
- 支持异步操作（接口请求、定时器等）
- 需手动加 deep: true 实现深度监听
- 典型场景：接口请求、表单校验、数据持久化（localStorage）

## 【问题】
Vue 模板编译过程？

## 【回答】
Vue 的模板编译，是把用户写的模板字符串转换成渲染函数的过程，**主要分为三步**：

**1. 解析（parse）**
使用解析器把模板字符串，通过正则等方式解析成 **AST 抽象语法树**，用 JS 对象来描述 DOM 结构。

**2. 优化（optimize）**
遍历 AST，**标记静态节点和静态根节点**。
这些节点不会发生变化，后续更新时可以跳过 diff，提升渲染性能。

**3. 代码生成（generate）**
把优化后的 AST 生成可执行的 **render 函数字符串**，再转成真正的渲染函数。
执行 render 函数就能生成 VNode，最终渲染成真实 DOM。

**简单说就是**：模板 → AST → 优化 AST → render 函数。

## 【问题】
Vue3 中 watch 和 watchEffect 的核心区别是什么？

## 【回答】
**一句话核心区别**

watch 是显式监听，需要指定监听的数据源；

watchEffect 是隐式监听，会自动收集依赖。

**详细版（面试直接说）**

**1. 依赖收集方式不同**

- watch 必须手动指定要监听哪个响应式数据，依赖明确。

- watchEffect 不需要手动指定，执行时自动收集函数内部用到的响应式数据。

**2. 初始化执行时机不同**

- watch 默认不执行，只有数据变化时才触发。

- watchEffect 会立即执行一次，再监听变化。

**3. 获取值不同**

- watch 可以同时拿到新值和旧值。

- watchEffect 只能拿到当前最新值，没有旧值。

**4. 使用场景不同**

- 需要精准监听、需要新旧值对比时，用 watch。

- 依赖多、逻辑复杂、需要初始化执行时，用 watchEffect。

**超精简版（30 秒背完）**

- watch：手动指定依赖，不立即执行，可获取新旧值。

- watchEffect：自动收集依赖，立即执行，只能获取最新值。

**简短版（面试官问：Vue 模板编译过程？）**
Vue 模板编译一共分为三个阶段：
1. **解析阶段**：把模板字符串解析成 AST 抽象语法树。
2. **优化阶段**：遍历 AST，标记静态节点和静态根节点，用于后续更新优化。
3. **代码生成阶段**：把 AST 转换成可执行的 render 渲染函数。
最终模板就变成了渲染函数，执行后生成虚拟 DOM。

**面试标准回答模板**：
Computed 是计算派生状态的，它会基于 reactive/ref 状态自动计算新值，并且有缓存，依赖不变时重复访问直接拿缓存值，适合用来做列表过滤、数据求和这类模板展示或逻辑计算；

Watch 是监听数据变化的副作用函数，数据一变就执行，支持异步、深度监听和立即执行，适合用来发接口请求、做表单校验或者把数据存到 localStorage。

**避坑提醒**：
- 不要用 Computed 做异步操作（比如发请求），它必须同步返回值
- 不要用 Watch 去计算展示值（比如列表总和），会重复执行且无缓存，性能差

## 【问题】
Pinia / Vuex 的核心概念？为什么需要集中式状态管理？

## 【回答】
Pinia 和 Vuex 都是 Vue 的集中式状态管理方案，其核心目的是在多个组件之间共享状态，并保证数据流清晰、可控。

核心概念包括 State、Getter、Action 和 Mutation（Vuex 中）。State 用于存储全局共享数据，Getter 用于派生状态，类似计算属性，Action 负责处理业务逻辑和异步操作，在 Vuex 中最终通过 Mutation 修改状态。Pinia 去除了 Mutation 的概念，直接在 Action 中修改 State，使模型更加简单直观。

集中式状态管理的价值在于避免多组件之间通过层层 props 或事件传递共享数据，统一管理全局状态，使数据来源清晰、修改路径可追踪，有利于大型中后台项目的维护和扩展。

**面试话术**：
在复杂应用中，状态往往会被多个组件依赖，集中式状态管理可以统一管理这些全局状态，减少组件耦合，提升可维护性。Pinia 相比 Vuex 设计更轻量，使用组合式 API，更符合 Vue3 的开发习惯。

## 【问题】
Vue 异步更新队列是如何工作的？

## 【回答】
Vue 的异步更新队列，核心是为了**性能优化**，避免频繁更新 DOM。

当我们修改响应式数据时，Vue 不会立即更新 DOM，而是将该组件的更新任务**缓存到一个异步队列**中。

如果同一事件循环内多次修改数据，Vue 会对队列进行**去重**，只保留最终状态。

等当前**所有同步代码执行完毕**后，Vue 会将队列里的更新任务作为**微任务**统一执行，**批量更新 DOM**。

整个机制基于 JS 的**事件循环**和**微任务**实现，保证更新高效且不阻塞渲染。

## 【问题】
为什么修改数据后立即访问 DOM 可能获取不到最新值？

## 【回答】
因为 Vue 更新 DOM 是**异步**的。

修改数据只会把更新任务推入异步队列，**不会立即执行 DOM 更新**。

我们立即访问 DOM 时，代码还处于**同步执行阶段**，DOM 还没被更新，所以拿到的是旧值。

必须等到同步代码执行完，Vue 才会在**微任务阶段**批量更新 DOM。

如果想获取最新 DOM，需要使用 `nextTick`，它会把回调放到 DOM 更新之后的微任务里执行。

## 【问题】
Vue3 的静态提升（Static Hoisting）和 Block Tree 机制如何优化渲染性能？

## 【回答】
Vue3 的静态提升和 Block Tree 机制，核心就是**减少虚拟 DOM 开销 + 跳过静态节点比对**，实现精准更新，分两部分说：

**1. 静态提升（Static Hoisting）**
编译阶段，把**完全静态的节点**提升到 `render` 函数外部，**只创建一次 VNode 并复用**，避免每次重新渲染都重复生成静态节点，从根源降低内存和渲染开销。

**2. Block Tree + Patch Flags**
编译时给**动态节点打上 Patch Flags 标记**，然后以 Block 为单元，把所有动态节点收集成一个**扁平化数组**。

更新时**直接跳过所有静态节点比对**，只遍历这个动态数组，结合标记做**靶向更新**，让虚拟 DOM 从全量递归 Diff 变成高效的精准更新。

## 【问题】
computed 缓存是怎么实现的？和 methods 有什么区别？

## 【回答】
计算属性的缓存，核心是依靠**依赖追踪 + 脏值标记（dirty）**实现的。

1. 当计算属性第一次被访问时，Vue 会执行它的 getter 函数，在执行过程中收集所有用到的响应式数据作为依赖。

2. 同时会给计算属性打上 **dirty 标记**，计算完后把 dirty 设为 false，**结果缓存起来**。

3. 之后再访问时，如果依赖没变化，dirty 还是 false，就**直接返回缓存结果，不再重新计算**。

4. 只有当它依赖的数据发生改变，才会把 dirty 重新设为 true，下次访问时重新计算并更新缓存。

和 methods 的本质区别是：

- **computed 有缓存，基于依赖变化才重新计算**；
- **methods 没有缓存，每次调用、每次渲染都会重新执行**。

**精简版（15 秒说完，适合紧张时）**
计算属性通过**依赖追踪**收集响应式依赖，再用 **dirty 脏标记**控制是否重新计算，实现惰性求值与缓存。

而普通方法**没有依赖收集和缓存**，每次调用都会执行，这是两者最本质的区别。

**加分版（面试官一听就知道你看过源码）**
Vue 内部会为每个 computed 维护一个 **watcher**，里面有个 **dirty** 字段。

初次求值时，会触发响应式数据的 getter 完成**依赖收集**，计算完把 dirty 置为 false。

当依赖更新时，依赖的 setter 会通知 watcher，把 dirty 重新置为 true。

下次访问 computed 时，**只有 dirty 为 true 才重新计算**，否则直接返回缓存值。

而 methods 只是普通函数调用，**没有依赖收集、没有缓存、没有脏检查**，每次调用都执行。

## 【问题】
什么是 Vue3 Vapor 模式？

## 【回答】

Vue3 Vapor 是 Vue 下一代**无虚拟 DOM**的渲染模式，它在**编译阶段**直接把模板编译成原生 DOM 操作，不再走虚拟 DOM 和 diff 流程，从而实现**更小的打包体积、更快的渲染速度、更精准的更新**，并且语法上兼容 Vue3。
你可以补一句：

核心逻辑完全一样，都是**依赖收集 + dirty 脏标记 + 缓存**。

只是 Vue2 用 `Object.defineProperty`，Vue3 用 `Proxy` 做响应式，但 computed 缓存机制没变。

## 【问题】
Vue 的事件系统是如何实现 v-on 的？

## 【回答】
Vue 的 v-on 事件指令底层分为**模板编译**和**运行时绑定**两个核心阶段，同时基于**原生事件**和**自定义事件**两套机制实现分发：

1. **编译阶段**：解析模板中的 v-on，提取事件名、回调函数和事件修饰符，生成对应的事件配置。

2. **运行时绑定**：根据绑定目标自动区分处理：
   - 绑定到**普通 DOM 元素**：直接调用浏览器原生 addEventListener 绑定原生 DOM 事件；
   - 绑定到**Vue 组件**：不操作 DOM，采用**发布订阅模式**，通过 $on 收集事件回调，等待子组件 $emit 触发自定义事件。

3. **事件触发**：原生事件由浏览器触发，自定义事件由组件内部 $emit 主动分发。
