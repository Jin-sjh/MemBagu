# HTTP 状态码

## 5. HTTP 状态码

### 【问题】
HTTP 状态码有哪些类别，分别代表什么意思

### 【回答】

**1xx（信息性状态码）：** 表示请求已被接收，继续处理。
- 例如：100 Continue

**2xx（成功状态码）：** 表示请求已成功处理。
- 例如：200 OK、201 Created

**3xx（重定向状态码）：** 表示请求需要进一步的操作（如跳转）。
- 例如：301 Moved Permanently、302 Found

**4xx（客户端错误状态码）：** 表示请求有错误，由客户端引起。
- 例如：400 Bad Request、404 Not Found、403 Forbidden

**5xx（服务器错误状态码）：** 表示服务器在处理请求时发生错误。
- 例如：500 Internal Server Error、502 Bad Gateway

## 详细状态码列表

### 【问题】
常见的 HTTP 状态码具体有哪些，各自的含义是什么

### 【回答】

**1xx 信息类**
- **100 Continue**：表示继续，一般在发送 POST 请求时，已发送了 HTTP header 之后，服务器端将返回此信息，表示确认，之后发送具体参数信息。

**2xx 成功**
- **200 OK**：表示正常返回信息，请求成功。
- **201 Created**：表示请求成功并且服务器创建了新的资源。
- **202 Accepted**：表示服务器已接受请求，但尚未处理。
- **204 No Content**：成功但无返回体。

**3xx 重定向**
- **301 Moved Permanently**：表示请求的网页已永久移动到新位置，永久重定向。
- **302 Found**：表示临时性重定向。
- **303 See Other**：表示临时性重定向，且总是使用 GET 请求新的 URI。
- **304 Not Modified**：表示自从上次请求后，请求的网页未修改过，使用缓存（协商缓存）。

**4xx 客户端错误**
- **400 Bad Request**：表示服务器无法理解请求的格式，客户端不应当尝试再次使用相同的内容发起请求。
- **401 Unauthorized**：表示请求未授权，未登录 / 认证失败。
- **403 Forbidden**：表示禁止访问，权限不足。
- **404 Not Found**：表示找不到如何与 URI 相匹配的资源，资源不存在。
- **405 Method Not Allowed**：请求方法不对。
- **429 Too Many Requests**：请求太频繁。

**5xx 服务器错误**
- **500 Internal Server Error**：表示最常见的服务器端错误，服务器报错。
- **502 Bad Gateway**：网关错误。
- **503 Service Unavailable**：表示服务器端暂时无法处理请求（可能是过载或维护），服务不可用。
- **504 Gateway Timeout**：网关超时。

## HTTP 失败状态码详解

### 【问题】
4xx 客户端错误状态码有哪些，分别代表什么意思

### 【回答】

**4xx（客户端错误）**
- **400**：请求参数 / 格式错误
- **401**：未登录、未授权
- **403**：权限不足，禁止访问
- **404**：接口 / 资源不存在
- **429**：请求过于频繁

### 【问题】
5xx 服务端错误状态码有哪些，分别代表什么意思

### 【回答】

**5xx（服务端错误）**
- **500**：服务器内部错误
- **502**：网关错误
- **503**：服务不可用 / 过载
- **504**：网关超时
- **429**：请求过于频繁

## HTTP 非 200 状态码排查方法

### 【问题】
浏览器输入 URL，如果响应不是 200，如何排查问题？

### 【回答】

响应非 200 状态码时，按「从客户端到服务器、从网络到业务」的顺序分层排查：

**第一步：先看状态码类型，定位大致范围**

- **1xx（信息响应）**：如 101 Switching Protocols（协议切换，常见于 WebSocket），需检查是否是正常协议升级，或确认服务器是否支持目标协议。

- **2xx（成功，但非 200）**：如 204 No Content（无响应体）、206 Partial Content（部分内容），需确认是否是预期行为（如文件分片请求），或接口逻辑是否返回了空数据。

- **3xx（重定向）**
  - **301/302/307/308**：检查重定向目标是否正确，是否存在重定向循环（如 A→B→A），用浏览器 Network 面板查看 Location 头。
  - **304 Not Modified**：说明资源未修改，检查缓存策略（Cache-Control/ETag）是否配置正确。

- **4xx（客户端错误）**
  - **400 Bad Request**：请求格式错误，检查 URL、参数、请求体是否正确。
  - **401 Unauthorized**：未认证，检查 Token、Cookie、Authorization 头是否缺失/过期。
  - **403 Forbidden**：无权限，检查用户角色、IP 白名单、Referer 防盗链配置。
  - **404 Not Found**：资源不存在，检查 URL 路径是否正确、服务器路由配置是否匹配。
  - **405 Method Not Allowed**：请求方法不支持（如用 POST 请求了只支持 GET 的接口），检查请求方法与接口定义是否匹配。

- **5xx（服务器错误）**
  - **500 Internal Server Error**：服务端代码报错，查看服务端日志。
  - **502 Bad Gateway**：反向代理/网关未找到后端服务（如 Nginx 转发到服务的连接失败）。
  - **503 Service Unavailable**：服务不可用（如服务宕机、重启、熔断触发）。
  - **504 Gateway Timeout**：网关超时，检查后端服务响应速度、网络连接是否正常。

**第二步：浏览器端基础排查**

- 打开「开发者工具 → Network 面板」，查看请求详情：
  - 确认请求 URL、Method、Headers、Payload 是否和预期一致。
  - 查看 Response Headers/Response Body，是否有错误提示、错误码、异常信息。
- 检查「Timing」（请求时间轴），确认是否是网络慢、DNS 耗时、TCP 连接慢、SSL 握手慢、TTFB（首字节时间）长等问题。
- 检查「Cache」标签，确认是否是缓存问题（如强缓存、协商缓存影响）。
- 尝试无痕模式打开，排除插件、扩展缓存的干扰。
- 换浏览器/换设备测试，确认是否是浏览器/设备的兼容性问题。

**第三步：网络层排查**

- 用 `ping` 或 `tracert`（Windows）/`traceroute`（Linux/Mac）检查域名解析和网络连通性。
- 用 `curl` 或 `telnet` 命令行发起请求，查看完整请求/响应流程，定位是请求发不出去还是响应有问题。
- 检查本地网络是否正常（如代理、VPN、防火墙是否拦截请求）。
- 检查是否是跨域问题（浏览器控制台会提示 CORS 错误，此时检查服务端是否配置了正确的 Access-Control-Allow-Origin 等响应头）。

**第四步：服务端/后端排查**

- 查看服务器访问日志（Nginx 日志、应用服务日志），定位错误发生的阶段（如请求到达 Nginx 但未到达应用服务，或应用服务抛出异常）。
- 检查服务器资源（CPU、内存、磁盘、网络带宽）是否正常运行。
- 检查反向代理/负载均衡配置，如 Nginx 的 proxy_pass 是否正确、负载均衡节点是否正常。
- 检查数据库、缓存、第三方依赖服务是否正常（如数据库连接失败导致接口报错）。
- 检查限流、熔断、WAF 拦截，确认是否被限流、WAF 拦截，查看相关拦截日志/监控。
